## 闭包的大小
闭包的大小取决于**实际在 Lambda 函数体内被使用（odr-used）的外部局部变量的数量**，而不是捕获说明符。

具体事实如下：

1.  **捕获机制实质**
    编译器会为 Lambda 生成一个匿名类（Functor）。对于按引用捕获的每一个变量，该类中会生成一个对应的成员变量，其类型通常是指针（在 64 位系统下即 `void*` 的大小）。

2.  **`[&]` 的行为**
    `[&]` 仅指示编译器：“如果我在 Lambda 内部使用了某个外部变量，请自动按引用捕获它”。它**不会**捕获当前作用域内的所有变量，只捕获被实际使用的变量。

3.  **大小计算逻辑**
    假设你在 Lambda 体内引用了 $N$ 个不同的局部变量（包括 `this` 指针）：
    *   $N = 0$：Lambda 为空类。根据 C++ 标准，空类大小至少为 1 字节（`sizeof` 通常为 1）。
    *   $N > 0$：闭包大小通常为 $N \times \text{sizeof(void*)}$。
        *   这取决于编译器的内存对齐（Padding）策略，但由于指针大小通常是对齐的，通常没有额外 Padding。

**结论：**

Lambda 闭包的大小相当于 $N$ 个 `void*`，其中 $N$ 是你在 Lambda 内部实际使用的外部局部变量的数量。

**示例（64位系统）：**

```cpp
void func() {
    int a, b, c;
  
    // 情况 1: 未使用任何变量
    // Size: 1 byte (空类)
    auto lambda1 = [&]() { }; 
  
    // 情况 2: 使用了 a
    // Size: 8 bytes (1个 void*)
    auto lambda2 = [&]() { a = 1; }; 
  
    // 情况 3: 使用了 a 和 b
    // Size: 16 bytes (2个 void*)
    auto lambda3 = [&]() { a = b; }; 
  
    // 情况 4: 使用了 a, b, c
    // Size: 24 bytes (3个 void*)
    auto lambda4 = [&]() { a = b + c; };
}
```
## 大闭包传参对比
```cpp
#include <chrono>
#include <iostream>
#include <numeric>
#include <vector>

// 跨平台禁止内联宏
#if defined(_MSC_VER)
#define NOINLINE __declspec(noinline)
#else
#define NOINLINE __attribute__((noinline))
#endif

// 1. 按值传递 (Pass by Value)
// 编译器需将 128 字节拷贝到栈上或寄存器中
template <typename F>
long long call_by_value(F f)
{
    // 防止编译器优化掉函数调用
    volatile long long result = f();
    return result;
}

// 2. 按引用传递 (Pass by Reference)
// 编译器仅传递 8 字节的指针，但函数内部访问捕获变量需双重寻址
template <typename F>
long long call_by_ref(const F& f)
{
    volatile long long result = f();
    return result;
}

int main()
{
    // 定义 16 个 long 变量 (16 * 8 bytes = 128 bytes for 64-bit pointers)
    volatile long v0 = 0, v1 = 1, v2 = 2, v3 = 3;
    volatile long v4 = 4, v5 = 5, v6 = 6, v7 = 7;
    volatile long v8 = 8, v9 = 9, v10 = 10, v11 = 11;
    volatile long v12 = 12, v13 = 13, v14 = 14, v15 = 15;

    // 创建捕获这 16 个变量的 Lambda
    // [&] 模式下，因实际使用了这些变量，闭包将包含 16 个指针
    auto large_lambda = [&]() -> long long {
        return v0 + v1 + v2 + v3 + v4 + v5 + v6 + v7 + v8 + v9 + v10 + v11 + v12 + v13 + v14 + v15;
    };

    std::cout << "Closure Size: " << sizeof(large_lambda) << " bytes" << std::endl;
    // 验证大小是否符合预期 (应为 128 字节左右)
    if (sizeof(large_lambda) < 128) {
        std::cout << "Warning: Compiler optimized storage. Test may be less accurate." << std::endl;
    }

    const int iterations = 100000000;

    // 测试 1: 按值传递
    auto start_val = std::chrono::high_resolution_clock::now();
    for (int i = 0; i < iterations; ++i) {
        call_by_value(large_lambda);
    }
    auto end_val = std::chrono::high_resolution_clock::now();

    // 测试 2: 按引用传递
    auto start_ref = std::chrono::high_resolution_clock::now();
    for (int i = 0; i < iterations; ++i) {
        call_by_ref(large_lambda);
    }
    auto end_ref = std::chrono::high_resolution_clock::now();

    auto duration_val = std::chrono::duration_cast<std::chrono::milliseconds>(end_val - start_val).count();
    auto duration_ref = std::chrono::duration_cast<std::chrono::milliseconds>(end_ref - start_ref).count();

    std::cout << "Iterations: " << iterations << std::endl;
    std::cout << "By Value Time: " << duration_val << " ms" << std::endl;
    std::cout << "By Ref Time:   " << duration_ref << " ms" << std::endl;

    if (duration_val < duration_ref) {
        std::cout << "Conclusion: Pass by VALUE is faster." << std::endl;
    } else {
        std::cout << "Conclusion: Pass by REFERENCE is faster." << std::endl;
    }

    return 0;
}
```
## 尤达体
不推崇。现代C++开发已不再使用且通常明确反对尤达条件（Yoda conditions，如 `if (1 == x)`）。

原因如下：

1. **编译器诊断能力取代了该写法的功能**
尤达条件的历史唯一作用是防止将相等判断（`==`）误写为赋值（`=`），例如将 `if (x == 1)` 误写为 `if (x = 1)`。在早期C/C++编译器中，后者是合法且不会报错的逻辑。
现代主流编译器（GCC、Clang、MSVC）在开启基础警告级别（如 GCC/Clang 的 `-Wall` 或 `-Wparentheses`）后，会直接对 `if (x = 1)` 抛出明确的警告（"suggest parentheses around assignment used as truth value"）。编译器层面的防御使这种代码层面的防御模式成为冗余。

2. **违反可读性原则**
程序代码的主语通常是变量而非字面量常量。`x == 1`（“如果x等于1”）符合线性逻辑认知，而 `1 == x`（“如果1被x等于”）增加了不必要的认知负荷。在防御误赋值的收益归零后，尤达条件只剩下降低代码可读性的负面作用。

3. **主流C++编码规范不支持或明确禁止**
- **C++ Core Guidelines**（C++核心指南）：不推荐该写法，指南要求依赖编译器警告和静态分析工具来捕获此类错误。
- **Google C++ Style Guide**：不使用尤达条件。
- **LLVM Coding Standards**：不建议使用尤达条件，认为主流编译器已经解决了误赋值问题。
- **WebKit C++ Style Guidelines**：明确禁止使用尤达条件。

4. **现代工具链与语法特性**
- 静态分析工具（如 Clang-Tidy、Cppcheck）会自动扫描并拦截控制流语句中的意外赋值。
- C++17 引入了带初始化的 `if` 语句（`if (init; condition)`），将变量作用域限制在条件分支内，进一步改变了条件判断的编写范式，强调清晰的条件表达式。

**结论**：
在现代C++开发中，应使用 `if (x == 1)`。防御意外赋值的正确方法是在构建系统中开启并强制把编译器警告视为错误（如 `-Wall -Wextra -Werror`），而不是使用尤达条件。若确实需要在 `if` 语句中进行赋值并判断，现代C++标准要求显式添加额外的括号：`if ((x = 1))` 以向编译器和阅读者申明这是蓄意行为。
