## 改名教训

* 大规模批量改名越是改得多，越要仔细看，不然暴雷
* 特别是与宏相关的，宏的参数可能既作为变量有作为字符串
* 变量改名要在理解其所作用范围的情况下去改动
  + 实际项目中可能会出现变量名与配置文件或二进制加密文件通过宏相绑定的问题
* 宏定义字面字符串改动也需要理解其作用范围
  + 可能与二进制文件内部信息对照

## 使英文字母小写变大写用英文怎么说？

* Convert lowercase to uppercase: 强调将小写字母转换为大写字母的操作。

## 性能优化使用版本

在进行性能优化时，必须以 Release 版（开启优化的版本）的性能数据为准。

以下是深度解析为什么不能以 Debug 版为准，以及两者在优化工作中的正确角色：

### 1. 为什么不能以 Debug 版为准？

*   优化器的“魔术”： 现代编译器（如 GCC, Clang, MSVC）在 `Release` 模式下会进行内联（Inlining）、循环展开（Loop Unrolling）、常量折叠、指令重排、向量化（SIMD）等。Debug 版通常关闭这些功能（`-O0`），此时的代码效率极低，甚至存在数量级的差异。
*   标准库实现的差异： 许多 C++ 标准库（尤其是 STL）在 Debug 模式下会加入大量的断言（Assert）、边界检查和迭代器有效性校验。例如，`std::vector` 的访问在 Debug 下可能比 Release 慢 10 倍以上。
*   掩盖真正的瓶颈： 在 Debug 版中，最耗时的往往是函数调用的开销和各种安全检查，而不是算法的核心逻辑。如果你针对 Debug 版进行优化，你可能是在优化那些在 Release 版中会被编译器直接“抹掉”的代码。
*   错误的结论： 你可能发现优化 A 在 Debug 版中提升了 50%，但在 Release 版中因为编译器已经自动做了类似处理，提升可能几乎为 0。

### 2. 什么时候看 Debug 版？

虽然不以它为准，但 Debug 版在优化过程中有其辅助作用：
*   逻辑正确性验证： 优化后的算法必须保证逻辑正确，Debug 版配合断言（`assert`）是验证逻辑的第一道防线。
*   内存调试： 使用 AddressSanitizer (ASan) 等工具通常在非完全优化版下更容易定位内存泄漏或越界。

### 3. 正确的优化工作流（Best Practices）

1.  在 Release 版下 Profile： 使用性能分析工具（如 `perf`, `VTune`, `Valgrind/Callgrind`, `Instruments`）对开启优化（-O2 或 -O3）并附带调试符号（-g）的可执行文件进行采样。
2.  建立基准（Baseline）： 记录 Release 版当前的耗时。
3.  实施优化： 修改代码或算法。
4.  再次测试 Release 版： 计算提速比（Speedup）。
    -   $Speedup = \frac{T_{Release\_Before}}{T_{Release\_After}}$
5.  防止“过度优化”： 现代编译器非常聪明。有时候你费尽心思写的汇编级优化，可能还没有编译器自动向量化的效果好。只有在 Release 版对比中才能发现这一点。

### 4. 特殊情况：RelWithDebInfo

在 CMake 流程中，推荐使用 `RelWithDebInfo` 模式。它等同于 `Release (-O2)` 但保留了调试符号（ `-g` ）。这样既保证了代码运行速度与生产环境一致，又能在性能分析工具中看到清晰的函数调用栈和代码行号。

### 总结

评估优化的最终KPI只有一个：Release 版快了多少。

如果一个优化方案在 Debug 版提升明显，但在 Release 版中无感，那么这个优化通常被认为是无效的，因为它并没有解决生产环境下的性能痛点。
